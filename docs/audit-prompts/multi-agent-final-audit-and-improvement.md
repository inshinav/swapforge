# SwapForge — мультиагентный финальный аудит и улучшение продукта

Скопируй весь промт ниже в новую задачу Codex с поддержкой команды агентов.

---

Ты — **Chief Product & Engineering Orchestrator**. Твоя задача — собрать и организовать большую команду сильных независимых специалистов, провести финальный глубокий аудит SwapForge, исправить подтверждённые проблемы и подготовить сервис к безопасному публичному запуску.

Ты отвечаешь за итог целиком. Не выполняй весь аудит в одиночку и не изображай команду текстом: создай реальных профильных агентов, назначь им конкретные ограниченные задачи, организуй лидов, контролируй зависимости, устраняй конфликты и лично проверяй ключевые выводы.

## Главная цель

После работы команды SwapForge должен:

- выглядеть современно, красиво, спокойно и минималистично;
- быть понятным обычному русскоязычному пользователю с телефона;
- вести человека от нуля до готового ролика по одному логичному действию за раз;
- стабильно создавать качественные AI-ролики с выбранной моделью, образом и объектами;
- до платного запуска выявлять плохой исходник, недостающие ракурсы и вероятные артефакты;
- поддерживать несколько одновременных генераций и видео до 60 секунд;
- правильно и безопасно принимать оплату через Lava.top и Crypto Pay;
- не терять проекты, результаты, платежи, баланс и фоновые задачи при ошибках или перезапуске;
- иметь понятную архитектуру, тесты, мониторинг и безопасный процесс релиза;
- давать результат, ради которого пользователь захочет вернуться и сделать следующий ролик.

Главный контрольный вопрос: **сможет ли новый пользователь с телефона без помощи владельца зарегистрироваться, оплатить, понять требования, получить хороший ролик и легко создать следующий?**

## Контекст продукта

- Канонический репозиторий: `https://github.com/inshinav/swapforge` (ветка `main`)
- Production: `https://inshinlab.com/swapforge/`
- Вход: Telegram, бот `@SwapForgeBot`
- Основная аудитория: RU-пользователи, преимущественно мобильные
- Оплата: Lava.top картой/СБП и Crypto Pay
- Пользовательские цены и баланс: USD
- Lava.top: RUB по фиксированному курсу `$1 = 100 ₽`
- Пополнение: произвольная сумма от `$5` до серверного максимума
- В пользовательскую цену всех AI-этапов включена скрытая маржа 25%; пользователь видит только финальную цену
- Максимальная длительность видео: 60 секунд; рекомендуемая — до 15 секунд
- Длинные видео делятся на части до 15 секунд и собираются в единый ролик
- Поддерживаются параллельные генерации
- Один пресет может содержать несколько фото модели, одежды и объектов
- Новая загрузка дополняет пресет, а не заменяет существующие фото
- Общие референсы имеют `variantId === null`
- В библиотеке сохраняются последние 20 роликов
- У владельца есть админка, балансы, активности и ручное начисление по Telegram-нику

## Неподвижные ограничения

1. Сначала проводится полный аудит и создаётся отдельный audit-коммит. Только после этого начинается исправление.
2. Не трать реальные деньги, не создавай настоящие платежи и не запускай платные AI-задачи без отдельного явного разрешения владельца.
3. Не публикуй, не деплой и не меняй production-конфигурацию до финального release gate и отдельного разрешения.
4. Не раскрывай токены, API-ключи, webhook secrets, cookies, персональные данные и значения production env.
5. Доктрина промтинга, системные промты, биллинг-инварианты и скрытая маржа не меняются без отдельного решения владельца. Их можно проверять, тестировать и оборачивать защитой.
6. Не ломай контракт `POST /api/billing/checkout`, CSRF, rate-limit и подписи webhook.
7. Не используй destructive git-команды. Сохраняй незакоммиченные пользовательские изменения.
8. Не разрешай двум агентам одновременно редактировать один файл или один модуль.
9. Каждый продуктовый коммит должен быть маленьким, тематическим и завершённым.
10. После каждого продуктового коммита обязательно должны пройти:

```bash
npm run typecheck && npm run lint && npm run test && npm run build
```

Точное число тестов фиксируется в отчёте и не должно уменьшаться без документированного удаления устаревшего теста и замены эквивалентным покрытием.

После отчётов лидов создай независимых проверяющих, которые не участвовали в первичном аудите:

Контрольная группа не добавляет общие пожелания. Она должна подтвердить или опровергнуть выводы первичной команды конкретными сценариями.

## Правила взаимодействия команды

1. Специалист отчитывается только своему лиду; лид удаляет дубли и передаёт консолидированный результат оркестратору.
2. Оркестратор назначает одного владельца каждой находки и каждого файла.
3. Read-only аудит можно выполнять параллельно. Редактирование выполняется только после freeze общего backlog.
4. Два специалиста не исправляют одну проблему независимо.
5. Если области пересекаются, назначается основной владелец и обязательные reviewers.
6. Если специалисты спорят, каждый предоставляет воспроизводимый тест или ссылку `файл:строка`; решение принимает лид, а P0/P1 подтверждает оркестратор.
7. Агент не может закрыть собственную находку. Исправление проверяет другой специалист.
8. Агент не объявляет релиз готовым по своей области, пока acceptance criteria не проверены фактически.
9. Любой lead может поставить release veto. Снять veto может только тот же lead после повторной проверки либо оркестратор на основании более сильного доказательства с записью причины.
10. Не скрывай неопределённость. Непроверенное отделяй от подтверждённого.

### Обязательная продуктовая проверка

Команда должна пройти путь:

1. лендинг;
2. Telegram-регистрация;
3. пополнение баланса;
4. короткое обучение;
5. создание базовой модели;
6. создание пресета с несколькими фото;
7. добавление образа и объекта, например мотоцикла;
8. загрузка исходного видео;
9. раскадровка;
10. проверка референсов и рисков;
11. показ общей финальной цены;
12. запуск генерации;
13. параллельное создание второго ролика;
14. обработка видео длиннее 15 секунд;
15. получение, просмотр и скачивание результата;
16. создание следующего ролика из сохранённого пресета;
17. работа библиотеки последних 20 роликов.

Для каждого шага ответь:

- понятно ли, что происходит;
- видно ли одно главное действие;
- можно ли ошибиться или застрять;
- понятна ли цена и последствия нажатия;
- сохраняется ли прогресс после reload/back/relogin;
- что увидит пользователь при медленной сети и ошибке;
- нет ли технического жаргона;
- сокращён ли путь до минимально необходимого.

### Обязательная UI/mobile matrix

Проверь 320, 375, 390, 768 и 1440 px. На 375 px для каждого экрана измерь:

```js
document.documentElement.scrollWidth <= document.documentElement.clientWidth
```

Проверь landing, login, onboarding, swap во всех состояниях, models/presets, library, billing, guide, admin и legal.

Проверь loading, empty, filled, validation, disabled, pending, success и error. Проверь safe area, экранную клавиатуру, touch-targets, контраст, focus, reduced motion, длинный русский текст, суммы и медиа.

### Обязательная проверка AI/video quality

- готовность модели учитывает `role='model'`, включая общие референсы `variantId === null`;
- один пресет содержит несколько фото и ролей;
- preflight проверяет нужные сценам ракурсы, масштаб, видимость, контакты рук/ног и важные детали;
- предупреждение объясняет, что переснять, какой ракурс добавить и какой риск останется;
- критические недостатки обнаруживаются до платного запуска;
- длинные видео корректно делятся на части до 15 секунд;
- границы выбираются там, где модель и важные объекты пригодны для continuity;
- следующий сегмент привязан к финальному состоянию предыдущего;
- fps, resolution, timebase, звук и overlap не дают скачка или потери длительности;
- ошибки сегмента, retry и сборка не создают ложный `done`;
- пользователь заранее видит цену всего ролика.

Проверь риски лица, тела, волос, одежды, шлема, мотоцикла, логотипов, рук, ног, контактов, дублей, морфинга, мерцания, света, фона, камеры и швов.

### Обязательная проверка нескольких генераций

Смоделируй минимум:

- две задачи одного пользователя;
- задачи двух разных пользователей;
- двойной click;
- reload во время работы;
- ошибку одного provider request;
- ошибку одного сегмента;
- перезапуск процесса;
- нехватку места;
- зависший polling;
- повторный callback.

Проекты, файлы, provider IDs, progress, holds, списания и результаты не должны смешиваться или дублироваться.

### Обязательная проверка платежей

- точные центы и одинаковые min/max на UI и сервере;
- Lava RUB = USD × 100;
- Crypto Pay проверяет asset, amount и status;
- подпись webhook проверяется до начисления;
- повтор webhook не начисляет второй раз;
- неверные amount/currency/asset/user не начисляют;
- checkout сохраняет CSRF, rate-limit и контракт;
- платёжная вкладка открывается синхронно, ошибка её закрывает, popup blocker использует текущую вкладку;
- pending сохраняется и проверяется на focus с backoff до примерно 10 минут;
- polling не работает без авторизации и останавливается после роста баланса;
- hold/settle/release атомарны;
- фактическое списание не превышает показанную цену;
- скрытая 25% маржа применяется ровно один раз;
- ручное начисление owner имеет request ID, автора, причину и audit trail.

### Обязательная security/infra проверка

Проверь Telegram signature/freshness/replay, sessions, CSRF, owner authorization, IDOR, XSS, SSRF, uploads, MIME spoofing, filename/path traversal, ffmpeg injection, webhook replay, secret leakage и tenant isolation.

Проверь nginx trusted proxy и rate-limit по реальному клиентскому IP. Один бот или пользователь не должен заблокировать вход и оплату всему сервису.

Проверь jobs, restart recovery, timeouts, retries, cleanup, storage rotation, backups, restore, systemd, nginx limits, health/readiness, deploy/rollback, логи, correlation IDs, метрики и алерты.

Приоритеты:

- `P0` — деньги, утечка, потеря данных, чужой доступ, массовая недоступность или невозможность основной функции;
- `P1` — массовый плохой результат, критический UX-тупик, нестабильная генерация или риск релиза;
- `P2` — заметное ухудшение качества, скорости, понятности или поддержки;
- `P3` — полезное улучшение без существенного риска запуска.

Лид обязан убрать дубли и отделить подтверждённые defects от гипотез.

## Фаза 3 — независимый challenge review

Контрольная группа получает отчёты без обсуждения с авторами и пытается:

- воспроизвести каждый P0/P1;
- найти пропущенную первопричину;
- доказать, что часть находок ложная или дублируется;
- проверить скрытые связи между UX, AI quality, billing и concurrency;
- пройти продукт как новый мобильный пользователь;
- атаковать tenant isolation, webhooks, двойные действия и restart recovery;
- проверить, не создадут ли предложенные исправления новые регрессии.

Создай `docs/final-team-audit/06-challenge-review.md`.

P0/P1 нельзя включать в финальный backlog без подтверждения вторым независимым специалистом. Неподтверждённые риски остаются отдельным списком.

## Gate A — утверждение аудита

Оркестратор создаёт `docs/final-team-audit/07-final-audit.md`:

1. вердикт `GO / GO WITH CONDITIONS / NO-GO` до исправлений;
2. краткое резюме для владельца;
3. карту системы;
4. journey map;
5. UI/mobile matrix;
6. quality/artifact matrix;
7. billing/security invariants;
8. подтверждённые P0→P3;
9. непроверенные риски;
10. результаты всех команд и число тестов;
11. единый backlog с dependency graph;
12. план implementation waves;
13. список замороженных областей, где нужно отдельное решение владельца.

Сделай отдельный коммит только с аудитом:

```text
audit: final multidisciplinary readiness review
```

Только после audit-коммита переходи к исправлениям. Если обнаружено решение, требующее изменения доктрины, биллинг-инвариантов, реального платежа, платной генерации или production, остановись и запроси разрешение.

## Фаза 4 — план исправлений

Оркестратор разбивает backlog на волны:

### Wave 1 — P0: безопасность, деньги и данные

Auth, tenant isolation, webhook/idempotency, баланс, holds, corruption, потеря файлов, массовая недоступность.

### Wave 2 — P1: основной путь и качество результата

Регистрация, оплата, onboarding, модели/пресеты, preflight, цена, запуск, progress, результат, длинные видео и параллельность.

### Wave 3 — P2: архитектура, устойчивость и наблюдаемость

Модульность, retries, recovery, cleanup, performance, logs, metrics, admin и тестовые пробелы.

### Wave 4 — P3: polish

Микротексты, анимации, мелкие визуальные улучшения и удобство постоянного пользователя.

Для каждой волны создай план с:

- порядком задач;
- владельцем;
- файлами;
- обязательными reviewers;
- тестами до изменения;
- acceptance criteria;
- rollback plan;
- зависимостями.

Одновременно редактировать можно только независимые области без общих файлов и контрактов.

## Фаза 5 — реализация

Правила каждого исправления:

1. Primary owner перечитывает находку и пишет/уточняет regression test.
2. Делает минимальное изменение без попутного рефакторинга.
3. Запускает релевантные тесты.
4. Передаёт изменение независимому reviewer.
5. Reviewer проверяет сценарий и ищет побочные эффекты.
6. Затем запускается полный gate:

```bash
npm run typecheck && npm run lint && npm run test && npm run build
```

7. Только после зелёного gate создаётся отдельный тематический коммит.
8. Следующая зависимая задача не начинается до успешного коммита.

Нельзя объединять в один коммит несвязанные UX, billing, security и infrastructure-изменения.

Если тест выявил старую проблему, не маскируй её, не ослабляй assertion и не удаляй тест. Верни проблему в backlog.

## Gate B — проверка после каждой волны

После каждой волны другой lead, не выполнявший исправления, проводит regression review:

- повторяет acceptance scenarios;
- проверяет mobile matrix;
- проверяет соседние критические потоки;
- сравнивает число и состав тестов;
- проверяет git diff на случайные изменения;
- подтверждает или отклоняет закрытие находок.

Создавай короткий wave report с закрытыми, открытыми и вновь найденными проблемами.

## Фаза 6 — финальная проверка release candidate

После всех исправлений проведи новый аудит не по diff, а по продукту целиком.

Обязательные участники release council:

- Chief Orchestrator;
- все пять лидов;
- Chief Challenger;
- First-Time User Reviewer;
- Security & Billing Red Team;
- Release Manager.

Release council повторно проверяет:

- путь от лендинга до первого результата;
- повторный ролик почти в один клик;
- все mobile widths и состояния;
- Telegram login;
- Lava/Crypto контракты без реального платежа;
- несколько одновременных генераций;
- длинный ролик до 60 секунд;
- ошибки, reload и restart recovery;
- tenant isolation;
- admin и ручное начисление;
- библиотеку последних 20 роликов;
- полный набор тестов и production build;
- отсутствие секретов и случайных файлов в diff.

Для визуального AI-качества используй только уже существующие результаты или отдельно разрешённую тестовую генерацию. Чётко отделяй визуально подтверждённое от вывода по коду.

## Финальные критерии запуска

Вердикт `GO` допустим только если:

- открытых P0 нет;
- открытых P1 нет либо владелец явно принял документированный ограниченный риск;
- security, billing, AI quality и SRE leads сняли veto;
- новый пользователь прошёл основной путь без помощи;
- на 375 px все экраны не имеют horizontal overflow;
- цена всегда видна до списания;
- duplicate actions/webhooks не создают двойных задач и денег;
- tenant isolation подтверждён;
- длинные и параллельные задачи не смешиваются;
- ошибки дают понятное действие и не теряют состояние;
- typecheck, lint, все тесты и build зелёные;
- rollback и диагностика подготовлены;
- production не изменён без разрешения.
